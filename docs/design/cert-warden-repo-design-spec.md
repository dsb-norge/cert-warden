> repository existed; imported at repo bring-up (July 2026). Some file paths and repository
> names it references are DSB-internal and not publicly accessible. Where this document and
> the implemented repository differ, the repository and its living docs under `docs/` win.

> Design spec for extracting Cert Warden (warden + sweeper + monitor) into its own **public,
> standalone repo in `dsb-norge`**, shipping composite actions and reusable workflows with a
> low-friction, highly automated CI/release system. Follows on from
> [`cert-warden-centralization-feasibility.md`](cert-warden-centralization-feasibility.md)
> (strategy C — hybrid — was accepted) and incorporates the owner's amendments: public in
> `dsb-norge`, ownership as for `github-actions-terraform`, CODEOWNERS deferred, monitoring
> optional with the Teams Notification Bot as the only supported channel.
>
> Grounded in: the current implementation in this repo (`.github/cert-warden/`,
> `.github/workflows/cert-*.yml`), the ops-repo origin copy, a structural review of
> `dsb-norge/github-actions-terraform` and `dsb-norge/github-actions`, the release setup of
> `dsb-norge/teams-notifier-function-app`, and July 2026 web research into release-please,
> GitHub immutable releases, bats-core/kcov/actionlint/zizmor, and how lego / certbot / traefik /
> acme.sh / terraform-provider-acme test ACME automation against Pebble. Sources in Appendix B.

> centralization immediately, superseding the feasibility doc's "after Phase 5" timing).

## 1. Purpose and scope

**The repo is the product**: a reusable, versioned suite that keeps Let's Encrypt certificates
for Azure public DNS zones renewed into Azure Key Vault, with optional monitoring and Key Vault
hygiene — consumed by any DSB landing zone as pinned GitHub Actions components.

In scope:

- The three tools as composite actions + packaged reusable workflows:
  - **warden** — enumerate delegated DNS zones, issue/renew via `lego` (ACME DNS-01, ARI-driven),
    import PFX to Key Vault, persist lego/ARI metadata, emit per-cert metrics.
  - **monitor** — evaluate a warden run's metrics against a lifetime-fraction SLO; optionally
    notify the DSB Teams Notification Bot.
  - **sweeper** — soft-delete orphaned/expired KV objects with dry-run and spike guards.
- A `setup-lego` action (versioned install + cache).
- The full test suite, CI, release automation, and documentation described below.
- Reference usage (canonical caller workflows) as part of the repo docs.

Non-goals:

- Provisioning the Azure side (KV, UAMIs, federated credentials, private endpoints, runners) —
  consumers own that via their Terraform; the repo documents the *requirements* only.
- Supporting notification channels other than the Teams Notification Bot (see §6.3).
- Supporting ACME CAs other than Let's Encrypt as a product goal (the ACME directory URL is
  overridable as a **test seam**, not a supported feature).
- Windows runners. Linux only (consumers run self-hosted Linux runners; CI uses `ubuntu-*`).

## 2. Repo name

**Decided: `dsb-norge/cert-warden`.** The suite has an established identity; the repo is a
product (scripts + actions + docs + local tooling), not a grab-bag of actions; product-named
repos are precedented in the org (`teams-notifier-function-app`, `azure-vm-image-builder`);
`uses: dsb-norge/cert-warden/...` reads best. (Alternatives considered and rejected:
`github-actions-cert-warden` — convention-fit but long and misfiles the non-action content;
`cert-warden-actions`; `azure-cert-warden`.)

Notes on deliberate deviations from `github-actions-terraform`:

- **Actions live under `actions/`**, not as top-level dirs — keeps the root readable for a repo
  that is also docs/tests/tooling. Same pattern as `grafana/shared-workflows`.
- **Reusable workflows are prefixed `reusable-`** to separate the consumer surface from the
  repo's own CI at a glance. (GitHub requires all of them directly in `.github/workflows/` —
  no subdirectories.)
- **One shared `lib/helpers.bash`** instead of N byte-identical copies — **decided** (was
  OQ-6). The "actions should be atomic" intuition is worth addressing head-on, because it turns
  out to be about *directories*, not *delivery*:
  - Delivery **is** atomic either way: `uses: org/repo/actions/x@<ref>` downloads the **entire
    repository archive** at that single ref — action and lib always travel together at the same
    SHA. There is no within-fetch version-skew scenario for a shared lib; that risk only exists
    across *refs*, which exact pinning (§8.2) already governs.
  - The only consumption mode a self-contained action directory would enable is copying the
    folder out of the repo — which is precisely the vendoring pattern this whole effort exists
    to eliminate. Not a supported mode; not worth designing for.
  - The duplication alternative has a *proven* cost: `github-actions-terraform` carries 24
    byte-identical `helpers.sh` copies held together by convention, and its own docs flag the
    drift risk.
  - Residual risk — GitHub changing action-fetch semantics to a partial/sparse download — is
    speculative (the full-repo download is documented behaviour), and is **continuously
    guarded** by the remote-fetch e2e (§9.7) on every PR: a regression is caught in this repo's
    CI before any consumer sees it. Contingency if it ever fires: a release-time "flatten" step
    that copies `lib/` into each action directory in the release commit — an implementation
    detail, not an interface change.

## 4. Scripts: required refactors and contracts

The scripts move essentially as-is (they are already fully env-var-parameterized), with these
changes:

### 4.1 Library mode, formalized

Replace the bespoke `CERT_WARDEN_LIB_ONLY=true` flag with the standard source-guard so every
script is bats-sourceable without ceremony:

```bash
# bottom of every script
if [[ "${BASH_SOURCE[0]}" == "${0}" ]]; then
  main "$@"
fi
```

This requires hoisting each script's top-level flow into a `main()` (warden's config region
stays top-level so sourcing loads config + functions; validation of required env vars moves into
`main()` / a `validateConfig()` so tests can source without exporting dummy config — removing
the boilerplate the current `selftest.sh` needs).

### 4.2 Prologue hardening

Every script starts with:

```bash
set -euo pipefail
shopt -s inherit_errexit   # command substitution inherits errexit (bash ≥4.4)
```

`inherit_errexit` is new — it closes the `$( )`-swallows-abort class the current selftest works
around with child-process probes. The bats suite keeps the child-process probe tests anyway
(regression guards for the documented `set -e` pitfalls, Appendix A).

### 4.3 Test seams (new, `CW_`-prefixed, default = production behaviour)

The integration harness (§9) needs three injectable seams. All are plain env vars, documented in
`docs/contracts.md` as **test seams — not supported for production use**:

| Env var | Default | Test harness sets |
|---|---|---|
| `CW_ACME_DIRECTORY_URL` | derived from `LE_ENVIRONMENT_NAME` (staging/production presets, unchanged) | `https://localhost:14000/dir` (Pebble) |
| `CW_LEGO_DNS_PROVIDER` | `azuredns` | `exec` (+ `EXEC_PATH` → challtestsrv hook) — lego's `azuredns` provider has no custom-endpoint knob, so the provider name is the seam; this is the same trade lego's own e2e makes |
| `CW_DNS_RESOLVER` | empty (system resolver for `dig`) | `@127.0.0.1 -p 5354` (CoreDNS test zone) — needed because the delegation check does `dig NS`, and challtestsrv cannot serve NS records |

Additionally lego trust env vars (`LEGO_CA_CERTIFICATES`, `LEGO_CA_SERVER_NAME`) pass through
untouched — the scripts must not filter the environment they hand to `lego`.

### 4.4 Stable env contract

The existing production env-var names (`AZ_TENANT_ID`, `AZ_SUBSCRIPTION_ID`, `AZ_DNS_RG_NAME`,
`AZ_CERT_KV_NAME`, `LE_*`, `CERT_*`, `KV_NAME`, `LOG_ONLY`, `MAX_DELETIONS`,
`TARGET_*`/`PROTECTED_PREFIXES`, monitor's `METRICS_FILE`/`BOT_*`/thresholds) are **kept
verbatim** so the two existing consumers migrate by swapping steps, not by re-plumbing. The
contract table lives in `docs/contracts.md` and is what SemVer protects (renames = major).

### 4.5 Runner-output hygiene (new requirements from research)

- Untrusted strings (ACME error bodies, zone names from Azure) are never echoed raw: wrap
  free-form multi-line output in `::stop-commands::<random-token>` guards, and multiline
  `GITHUB_OUTPUT` writes use random heredoc delimiters (delimiter-injection class,
  CVE-2022-35954).
- Large payloads (PEMs, `az` JSON) go through files in `$RUNNER_TEMP`, never env vars — with
  `set -o allexport` in the loader pattern this is what keeps us clear of the 128 KiB
  per-var / ~2 MiB ARG_MAX ceiling that repeatedly bit `github-actions-terraform`.

## 5. Public interface

### 5.1 Composite actions

Thin `action.yml` shims mapping inputs → env, then sourcing the step script — the modern
`github-actions-terraform` shape. **Every `run:` in every action declares `shell: bash`
explicitly** (mandatory in composite actions; also pins the `-eo pipefail` wrapper).

**`actions/setup-lego`** — inputs: `lego-version` (default: the repo's pinned known-good, bumped
by Renovate); installs via `go install github.com/go-acme/lego/v5@<version>` with
`actions/cache` keyed on version+os+arch, appends to `GITHUB_PATH`. Output: `lego-path`.

**`actions/warden`** — inputs mirror the env contract (`tenant-id`, `subscription-id`,
`dns-rg-name`, `key-vault-name`, `le-environment`, `le-account-email`, `force-all-new`,
`force-renewal`, `tag-application-name`, `tag-created-by`, `tag-description`,
`metrics-output-file`). Assumes `az login` and lego already done (caller/reusable workflow's
job). Outputs: `metrics-file`, `zones-total`, `zones-failed`, `summary-json` (compact per-run
aggregate).

**`actions/monitor`** — inputs: `metrics-file`, `environment-label`, `warn-threshold`,
`page-threshold`, `liveness-window-hours`, `warden-conclusion`, `warden-run-url`,
`metrics-age-hours`, `force-notify`, `dry-run`, and the bot triple `bot-api-base`,
`bot-api-audience`, `bot-alias` — **all three optional; when absent the action runs in
evaluate-only mode** (see §6.3). Outputs: `severity`, `min-lifetime-fraction`, `managed-count`,
`failed-count`, `worst-zone`, `reasons-json`, `notified` (bool), `notify-http-status`.

**`actions/sweeper`** — inputs: `key-vault-name`, `log-only` (default `"true"`),
`max-deletions`, `target-cert-prefixes`, `target-secret-prefixes`, `protected-prefixes`.
Outputs: `candidates-count`, `deleted-count`, `candidates-json`.

### 5.2 Composite-action input hygiene

Composite inputs are **not** exposed as `INPUT_*` env vars; each shim maps them explicitly via
`env:` — never interpolating `${{ inputs.x }}` into script bodies (the injection-safe
indirection pattern; enforced by zizmor's `template-injection` audit in CI).

### 5.3 Reusable workflows (the packaged shape)

Each wraps one job with the canonical step sequence; consumers own triggers, matrix, and
concurrency (irreducible per GitHub's model — see feasibility §3.1).

**`reusable-warden.yml`** — inputs: `environment` (label), `runs-on`, `azure-tenant-id`,
`azure-subscription-id`, `azure-client-id`, `key-vault-name`, `dns-rg-name`, `le-environment`,
`le-account-email`, `lego-version` (optional), `force-all-new`, `force-renewal`, tag values,
`metrics-artifact-name` (default `cert-warden-metrics-<environment>-<run_id>-<run_attempt>`),
`metrics-retention-days` (default 90). Steps: `azure/login` (OIDC) → `setup-lego` → `warden` →
`upload-artifact` with `if: always()`. Workflow-level outputs re-export the warden action's
outputs plus `metrics-artifact-name`. Required permissions documented in the header:
`id-token: write`, `contents: read`.

**`reusable-monitor.yml`** — inputs: `environment`, `runs-on`, `monitor-client-id` (optional —
only needed for notify mode), `azure-tenant-id`, `warden-workflow-name` (default
`"Cert Warden"`; the `workflow_run`-vs-latest-completed resolution logic — including the
skip-`skipped`-runs semantics — lives here, centralized), threshold/liveness inputs, the
optional bot triple, `force-notify`, `dry-run`. Steps: conditional `azure/login`
(`allow-no-subscriptions: true`, only when bot inputs present) → resolve warden run → download
metrics artifact (`continue-on-error` — absence is itself a signal) → `monitor`. Re-exports all
monitor outputs. Permissions: `id-token: write`, `contents: read`, `actions: read`.

**`reusable-sweeper.yml`** — inputs: `environment`, `runs-on`, `azure-tenant-id`,
`azure-subscription-id`, `azure-client-id`, `key-vault-name`, `log-only` (default `true`),
`max-deletions`, prefix overrides. Re-exports sweeper outputs. The doc header states the
concurrency contract: **callers must put sweeper and warden in the same concurrency group per
environment** (they mutate the same vault); reference usage shows it.

Design rule to prevent `terraform-ci-cd-default.yml`-style sprawl: **workflow inputs are
configuration, not behaviour flags.** A consumer needing different behaviour composes the
actions in their own workflow instead (the hybrid escape hatch). New inputs require a
justification line in the PR description.

### 5.4 Versioned contracts

Three cross-component contracts are part of the public API and covered by SemVer:

1. **The env-var contract** (§4.4) — `docs/contracts.md`.
2. **The metrics artifact**: JSON array of per-zone records; formal JSON Schema at
   `contracts/metrics.schema.json`, including a new `schema_version` field per record. Unit
   tests validate `recordCertMetric` output against the schema; the monitor's tests consume
   schema-generated fixtures. Additive fields = minor; renames/removals = major.
3. **The KV object naming scheme** (`le-cert-<le-env>-<zone-dashed>-pfx`,
   `letsencrypt-<le-env>-account-*`, `*-meta` ARI secrets) — this is *state in every consumer's
   vault and the name their AGW reads*; treated as the most breaking-averse contract of all.

## 6. Monitoring stance (per owner decision)

1. **Monitoring is optional.** It's a separate action + reusable workflow; consumers who don't
   want it simply don't call it. Nothing in warden depends on monitor.
2. **If used, the DSB Teams Notification Bot is the supported channel** — the payload
   (Adaptive Card), auth (OIDC → `az account get-access-token --resource <bot-api-audience>`),
   and endpoint (`POST {bot-api-base}/v1/notify/{alias}`) match
   `dsb-norge/teams-notifier-function-app`'s API. No pluggable notifier abstraction is built or
   documented.
3. **Generous outputs enable roll-your-own**: because the monitor action/workflow exports the
   full evaluation (`severity`, `reasons-json`, counts, `min-lifetime-fraction`) and works in
   evaluate-only mode without the bot triple, a consumer can bolt any delivery step onto the
   outputs. The reference docs mention this in one sentence and explicitly mark it unsupported —
   matching the "be generous in outputs, don't spend effort supporting alternatives" directive.
4. **Raw Adaptive Cards are first-class in the bot API** (verified against
   `teams-notifier-function-app`'s API reference, resolving OQ-9): `POST /v1/notify/{alias}`
   accepts `format: "adaptive-card"` with `message` as **the raw Adaptive Card JSON object**
   (the doc's own example is a version-1.4 card), plus a string-valued `metadata` map —
   exactly what the current `monitor.sh` already emits. So the suite keeps authoring its own
   cards and posting them raw; no bot-side card builder constrains us. Compatibility is checked
   during testing (§9.4): the bot-sink assertions validate the emitted body against the
   documented request schema (`format` field, card object shape with `type`/`version`/`body`,
   string-only metadata values) and watch the error surface the API documents
   (400/413/415/429 semantics — notably 413, since FactSets over many zones grow). The bot API
   remains an external contract owned by `teams-notifier-function-app`; this repo pins the
   request shape one-sidedly and revisits only on a breaking bot-API change.

## 7. Reference usage (shipped as repo docs)

`docs/reference-usage.md` contains complete, copy-pasteable caller workflows — the standard way
of consuming — kept current by release-please annotation bumps (§8.2) so the pinned refs in the
examples always match the latest release:

1. **Full suite** (the recommended shape): three caller workflows mirroring today's env-config
   setup — warden (schedule + dispatch + `workflow_run` after deploy; per-env matrix with
   `max-parallel: 1` and `fail-fast: false`; concurrency group
   `cert-warden-maintain-certificates-<env>`), monitor (`workflow_run` after warden + daily
   liveness cron + dispatch with `force_notify`/`dry_run`), sweeper (dispatch with `log_only`
   defaulted true + optional weekly cron, sharing the warden concurrency group).
2. **Warden only** (no monitoring, no sweeper) — the minimum viable consumer.
3. **Evaluate-only monitoring** — monitor without the bot triple, wiring outputs into a
   consumer-owned step (the one-sentence unsupported example).

All examples pin `@<full-sha> # vX.Y.Z` and note the floating `v1` alternative for
lower-friction tracking (§11.1).

### 7.1 Sweeper defaults and the path to full auto (per owner decision)

Graduation ladder, documented in reference usage:

1. **Step 0 (default)**: caller has `workflow_dispatch` only; `log_only` input defaults `true`.
   Dispatch → review the `candidates-json` output / step summary.
2. **Step 1**: dispatch once with `log_only: false` — validated destructive run,
   `max-deletions` spike guard active.
3. **Step 2 (set-and-forget)**: enable the weekly cron in the caller with `log_only: false`.
   The reference caller ships with the cron block present but commented, exactly like
   env-config's rollout did — the "full auto" end state is a two-line uncomment.

### 7.2 Caller-side LE-staging canary — prototype (per owner decision, evaluate later)

`docs/reference-usage-canary.md` ships **marked PROTOTYPE / EXPERIMENTAL**: a caller-repo
workflow shape that gives consumers an early-warning signal for upstream change (LE policy
shifts, ACME/ARI behaviour changes, lego regressions) *between* suite releases. It must live in
a calling repo — it needs real Azure DNS, a real UAMI, and a network path — this repo's CI
cannot run it. Shape:

- A scheduled caller (weekly suggested) invoking `reusable-warden.yml` with
  `le-environment: staging` and `force-all-new: true` against the consumer's existing zones —
  a full issuance exercised end-to-end against **real LE staging** every cycle, independent of
  whether production renewals happen to be due.
- Followed by `reusable-monitor.yml` in notify mode against a low-noise alias — alerting the
  *caller* when the canary run fails or degrades (this is the "alert callers of coming
  pitfalls" channel).
- Cleanup is free by design: staging artifacts land as `le-cert-staging-*` objects, which are
  exactly what the sweeper's default target prefixes reap — a consumer running the full suite
  gets canary hygiene for nothing.
- Rate-limit note: LE **staging** limits are far above production's; a weekly canary over a
  typical zone count is negligible (documented with current numbers in the doc).

Keep-or-delete is deliberately deferred: the doc ships as a prototype, we run it in the first
consumer for a while, then jointly evaluate whether it becomes a supported reference shape or
gets dropped (D-21).

### 7.3 `consumer-prerequisites.md` — requirements with example declarations, zero private links

Content: the UAMI + RBAC set (KV Certificates/Secrets Officer, the custom DNS TXT role),
federated credential (subject `repo:<org>/<repo>:ref:refs/heads/main` — and why `main`-only),
KV expectations (RBAC mode, network path from the runner), runner requirements, LE account
email policy, LE staging-first onboarding sequence, and production rate-limit planning notes.

Form (per owner decision): **describe with self-contained example Terraform declarations**
(`azurerm_user_assigned_identity`, `azurerm_federated_identity_credential`,
`azurerm_role_assignment`, custom `azurerm_role_definition` for DNS TXT) written for the doc —
**never** by linking to the private implementations. This is a hard, repo-wide rule (§12): **no
links to private DSB repos anywhere in the public repo** — docs, code comments, commit messages
(they surface in the generated changelog), PR/issue templates, release notes. Internal
provenance, where genuinely needed (e.g. the fresh-start note, D-24), is expressed as generic
wording ("derived from DSB-internal automation"), not names-with-links. CI enforces the
mechanical part (§9.1: private-reference guard).

## 8. Release engineering

### 8.1 Flow

- **release-please** (`googleapis/release-please-action@v5`, pinned by SHA),
  `release-type: simple` (maintains `version.txt` + `CHANGELOG.md`) with
  `release-please-config.json` + `.release-please-manifest.json` — the same shape
  `teams-notifier-function-app` already uses, so the org knows how to drive it.
- Pushes to `main` accumulate into a single open **release PR**; merging it creates tag
  `vX.Y.Z` + a GitHub Release. Conventional commits drive the bump
  (`fix:` patch / `feat:` minor / `!`+`BREAKING CHANGE:` major; `Release-As:` override).
- **Same-run post-release steps**, gated on `steps.release.outputs.release_created` (avoids the
  GITHUB_TOKEN-doesn't-trigger-workflows trap entirely):
  1. Move the floating major tag: `git tag -f v1 && git push -f origin v1` (plain annotated tag).
  2. Nothing else — no artifacts to build; the release *is* the tag.

### 8.2 Internal ref pinning (the anti-`@v0` mechanism)

Reusable workflows reference sibling actions with **exact pins + release-please generic-updater
annotations**:

```yaml
- uses: dsb-norge/cert-warden/actions/warden@v1.2.3 # x-release-please-version
```

`release-please-config.json` lists each reusable workflow (and `docs/reference-usage.md`,
`README.md`) as `{"type": "generic", "path": ...}` extra-files — the `"generic"` type is
**mandatory** for `.yml` paths (the extension-based default picks the YAML updater, which can't
do in-line substring replacement). Rules the repo docs must state: exactly **one semver string
per annotated line**; the `v` prefix is preserved; block markers
(`x-release-please-start-version` / `x-release-please-end`) for multi-line example snippets in
docs. Precedent: `grafana/shared-workflows` (README usage snippets) and `super-linter`
(`action.yml` image tag) do exactly this.

Consequence: a consumer pinning `reusable-warden.yml@v1.2.3` gets actions at `v1.2.3` —
behaviour actually frozen, unlike `github-actions-terraform`'s floating internal `@v0` refs, and
**no dev-tag find/replace ritual ever** (§9.6 replaces it).

### 8.3 Token: GitHub App (hard requirement)

Because the release PR edits files under `.github/workflows/`, `GITHUB_TOKEN` **cannot** be used
(GitHub refuses workflow-file modification without the `workflows` permission, which the
workflow `permissions:` block cannot grant). The release workflow therefore uses a **dsb-norge
GitHub App** token (`actions/create-github-app-token`) with repository permissions:
`contents: write`, `pull-requests: write`, `issues: write`, `workflows: write`. A side benefit:
App-created release PRs trigger CI normally (the GITHUB_TOKEN-created ones would not), so the
release PR itself is fully tested before merge. Also enable the repo setting *Allow GitHub
Actions to create and approve pull requests*. Whether to reuse an existing org App or mint a
dedicated one is OQ-1 (owner investigating; most probably a new App scoped to the repo).

### 8.4 Immutability and tag governance

- **Enable GitHub immutable releases** (GA since 2025-10-28) from day one: every `vX.Y.Z`
  Release is immutable (assets + tag locked, Sigstore attestation for free).
- **Never create a Release object for `v1`** — the floating major must stay a plain git tag or
  immutability freezes it (GitHub's own actions-versioning doc is explicit on this split).
- **Tag ruleset** on `v*`: block update/delete; bypass actor = the release App only.
  `preview/*` tags (§9.6) are excluded from the ruleset and never get Releases.
- Renovate in this repo sets `ignoreDeps: ["dsb-norge/cert-warden"]` so the bots never race
  release-please on the annotated internal refs.
- Repo Renovate config extends the org presets (incl. `helpers:pinGitHubActionDigests` for
  third-party actions) plus a **custom regex manager** for the tool versions embedded in
  defaults: `lego-version` in `actions/setup-lego/action.yml` and the Pebble image tags in
  `tests/harness/docker-compose.pebble.yml` — so even the test stack is bot-maintained.

### 8.5 Commit convention — merge commits, commitlint-gated (decided)

Conventional commits are load-bearing (they drive release-please). Per owner decision the org
convention holds: **merge commits, never squash** — release-please supports this provided
*every* commit on main parses. CI therefore gates every PR with **commitlint** over the full PR
commit range plus the PR title.

**Developer-experience requirements for the gate** (per owner: informative errors, not just a
red X):

- On failure, the commitlint job **upserts a sticky PR comment** (same mechanism as the preview
  comment, §9.6) containing: a table of offending commits (short SHA, subject, the exact rule
  violated, what a passing version looks like), a compact cheat-sheet of the allowed types with
  one-line examples (`feat: …` → minor, `fix: …` → patch, `feat!:`/`BREAKING CHANGE:` → major,
  `chore/docs/test/ci/refactor: …` → no release), and the fix procedure — since the repo never
  squashes, that's `git rebase -i` + `reword` with the exact commands spelled out, plus a link
  to the relevant `development-and-ci.md` section.
- The comment is removed/updated to a green confirmation once the range passes, so PRs don't
  accumulate stale nagging.
- Failures also surface as file-level annotations on the run (native commitlint action
  behaviour) — the comment is the *explainer*, annotations are the *pointer*.
- Prevention beats correction: `scripts/install-git-hooks.sh` installs a local `commit-msg`
  hook running the same commitlint config (no husky/node ceremony beyond what commitlint itself
  needs), and `development-and-ci.md` shows it in the getting-started steps.
- Tool choice at implementation time (`wagoid/commitlint-github-action` + a comment step, or a
  small bespoke job); the config (`commitlint.config.mjs`, conventional preset + the
  no-private-URL rule from §12) and the behaviour above are the spec.

## 9. CI and test architecture

Design goals, restated from the owner: **much lower friction than `github-actions-terraform`**
(no dev-tag swap, no manual test ritual), a real test pyramid with high coverage, mock only at
unavoidable boundaries, PR = everything you need to test from a calling repo, main never carries
test scaffolding, and the whole thing documented in-repo.

**All CI runs on GitHub-hosted runners.** Public repo + self-hosted runners is a categorical
no (GitHub security guidance); nothing in CI needs Azure or DSB networks — that's what the
harness below is for.

### 9.0 Layer overview

| Layer | What | Runs | Gate? |
|---|---|---|---|
| L0 static | shellcheck, shfmt, actionlint, zizmor, yamllint, pinact, commitlint | every PR, seconds | required |
| L1 unit | bats-core, sourced functions, shims/shellmock | every PR, seconds | required |
| L2 integration | real lego ↔ Pebble/challtestsrv; az shim; CoreDNS; bot sink | every PR, ~1-2 min | required |
| L3 self-test | `uses: ./actions/*` + local reusable workflows, OS matrix + minimal container | every PR | required |
| L3b remote-fetch e2e | dispatch `preview-consume.yml` at the preview tag | every PR (same-repo) | required |
| L4 canary | staged consumer rollout via version-bump PRs | post-release | consumer-side |
| coverage | kcov over L1+L2 → local `jq` threshold | every PR | advisory initially (§9.8) |

A single **`ci-conclusion`** job needs-aggregates everything — the one required status check
for branch protection (same pattern as `github-actions-terraform`'s `tests-conclusion`, which
worked well there).

### 9.1 L0 — static

- **shellcheck** with `.shellcheckrc` (`external-sources=true`, `source-path=SCRIPTDIR`);
  covers `.bats` files too (shellcheck ≥ 0.7). Env-var config documented at declaration sites
  with `: "${VAR:?required}"` / targeted `# shellcheck disable=SC2034  # consumed by X` — no
  blanket disables.
- **shfmt** `-d` (reads `.editorconfig`; `-ln bats` for test files).
- **actionlint** — validates `${{ }}` expression types, `with:` keys against each `action.yml`,
  local reusable-workflow call targets, cron syntax; runs shellcheck over inline `run:` blocks.
- **zizmor** with SARIF upload to code scanning — template-injection, excessive-permissions,
  artipacked, cache-poisoning, unpinned-uses audits.
- **yamllint** (`truthy: {check-keys: false}`).
- **pinact --check** — enforces SHA-pinning + version comments across workflows *and*
  action.yml files (chosen over ratchet for composite-action coverage).
- **commitlint** (per §8.5, with the sticky-comment explainer).
- **private-reference guard** — a deny-list grep over the working tree, commit messages in the
  PR range, and the generated changelog for private-org URL patterns (the
  deny-list lives in `.github/private-ref-patterns.txt`). Enforces the §7.3/§12 never-link-private rule
  mechanically, including the subtle path where a commit message referencing a private repo
  would otherwise surface in the public `CHANGELOG.md`.

### 9.2 L1 — unit (bats-core)

- **bats-core** pinned via the official `bats-core/bats-action` (installs bats +
  support/assert/file); suites under `tests/unit/`, one per script, `--jobs` parallel,
  `--formatter junit` for PR annotation.
- Scripts are sourced directly (source-guard, §4.1) and functions exercised with
  **bats-assert** assertions. External commands are mocked per-test with
  **boschresearch/shellmock** (argument-matching + call assertions — scales better than
  bats-mock's strict invocation plans for a script that calls `az` dozens of times) or plain
  PATH shims for simple cases.
- The current `selftest.sh` probes are **ported, not discarded**: the
  `((count++))`-under-`set -e` regression guard and the metrics-on-failure guard become bats
  tests that still run their probe in a **child bash process** (command substitution masks
  errexit aborts) under the exact runner wrapper (`bash --noprofile --norc -eo pipefail`).
  The pitfalls catalog (Appendix A) each get a named regression test where applicable — the
  "improved and developed selftest pattern" the owner asked for is precisely this: every
  documented pitfall pinned by a test.
- Known bats gotchas honored in `tests/test_helper.bash`: `run` + pipes wrapped in functions,
  fd-3 closing for background processes (the monitor's curl), no `@test` in loops.

### 9.3 L2 — integration (the real-as-possible layer)

The mock boundary, derived from the actual call surface and the research findings:

| Dependency | Real or fake | How |
|---|---|---|
| ACME protocol + issuance | **REAL** | Pebble (`ghcr.io/letsencrypt/pebble`), `-strict` |
| lego (client, ARI, retries) | **REAL** | the pinned production binary via `setup-lego` |
| DNS-01 challenge write | fake at the provider seam | lego `--dns exec` → `challtestsrv-hook.sh` → challtestsrv mgmt API (`/set-txt`, `/clear-txt`) — lego's own e2e pattern |
| DNS propagation check | **REAL** (against test DNS) | `--dns.resolvers 127.0.0.1:8053` + `--dns.propagation-rns` |
| Delegation check (`dig NS`) | **REAL** (against test DNS) | CoreDNS `file`-plugin container serving the test zone with NS records (challtestsrv can't serve NS), via `CW_DNS_RESOLVER` |
| PFX assembly/validation | **REAL** | production `openssl` paths; harness verifies the produced PFX chains to Pebble's per-boot root (fetched from `:15000/roots/0`) |
| `az` CLI | **fake** (unavoidable) | PATH shim (below) |
| Key Vault | via the shim's state dir | no emulator is both certificate-capable and az-compatible (LocalStack-Azure lacks the certificates API; Lowkey Vault targets SDKs; `az` cannot mint tokens against fakes — Azure/azure-cli#31958) |
| Teams bot | **fake** HTTP sink | asserts method/path/auth-header shape + Adaptive Card payload against the schema |
| jq, curl, coreutils | **REAL** | — |

**The `az` shim** (`tests/harness/az-shim/az`): a dispatching bash fake covering the ~12
subcommands the suite calls (`network dns zone list`, `network dns record-set ns show`,
`keyvault secret show|set|list|delete`, `keyvault certificate show|list|import|delete`,
`account set`, `account get-access-token`). It keeps state in a per-test directory (secrets +
certs as files with attributes JSON), appends every invocation to a call log for assertions,
and serves fixture JSON **captured once from real `az` output** (fixture fidelity is the main
risk of CLI stubbing; captured-not-handwritten is the mitigation). Its
`keyvault certificate import` handler performs a *real* `openssl pkcs12` parse and chain
verification against the Pebble root — the mock boundary itself validates the artifact, the
trick that keeps a stubbed layer honest.

**Pebble wiring**: vendored `tests/harness/docker-compose.pebble.yml` (the upstream compose
doesn't publish the DNS port to the host) — pebble with
`-strict -dnsserver 10.30.50.3:8053`, challtestsrv publishing `8055` (mgmt) and `8053` tcp+udp
(Pebble ≥ 2.10 queries DNS over TCP). Not GHA `services:` — service containers can't override
the container command, which this wiring needs; no surveyed project uses `services:` for Pebble.
Determinism: `PEBBLE_VA_NOSLEEP=1`, `PEBBLE_WFE_NONCEREJECT=0` — **plus one dedicated
chaos test with `PEBBLE_WFE_NONCEREJECT=20`** (lego's own e2e value) proving the retry path,
and a `set-servfail` negative test proving a zone failure produces a `failed` metric record and
a non-zero exit *after* the loop completes (the exact bug class that motivated the selftest).

Scenario coverage (each a bats file): first issuance; ARI-driven renewal (metadata persisted and
re-read through the shim); SAN mismatch → reissue; non-delegated zone skipped; partial failure →
metrics still emitted + exit code correct; account reuse from KV vs new registration; sweeper
dry-run/real/spike-guard/protected-prefix on a seeded state dir; monitor severity matrix from
schema-valid fixtures + notify against the sink + evaluate-only mode.

**Rate limiting — explicitly a non-concern in CI, a consumer concern in production.** Pebble's
README is unambiguous: *"Pebble does not enforce any rate limits. It is not presently an
appropriate tool for testing that your client handles Boulder/Let's Encrypt rate limits
correctly."* So the CI suite can run arbitrarily often and parallel with zero rate-limit risk —
and, flip side, it **cannot** simulate an LE `rateLimited` ACME error. That gap is acceptable:
from the warden's perspective a rate-limit rejection is just another failed issuance (lego
surfaces the ACME error → `failed` metric record → monitor evaluates), i.e. the generic
failure-path scenarios already cover the *handling*; only the specific trigger is untestable via
Pebble. Real LE limits live where real LE is touched: consumer production runs (rate-limit
planning in `consumer-prerequisites.md`, §7.3) and the staging canary (§7.2 — staging limits are
far above any weekly canary's usage).

### 9.4 Contract tests

- Metrics records validated against `contracts/metrics.schema.json` in both directions
  (producer output validates; monitor fixtures generated from the schema).
- The bot request contract (path/auth/payload — raw Adaptive Card per §6.4) pinned in the sink
  assertions, referencing the `teams-notifier-function-app` API doc version.

### 9.5 L3 — workflow self-tests (same-commit)

`ci.yml` jobs exercising the shipped surfaces exactly as consumers will:

- Each composite action invoked via **`uses: ./actions/<name>`** (actions/checkout first) in
  scenario jobs with asserted outputs — the actions/checkout `test.yml` pattern.
- Each reusable workflow called via **`uses: ./.github/workflows/reusable-*.yml`** — local
  same-commit semantics, so the PR's version is what runs. The Azure-login step is skipped in
  harness mode via an internal `test-mode` mechanism (an undocumented input or, cleaner, the
  login step keyed off `azure-client-id != ''` — evaluate during implementation; whichever is
  chosen, harness mode must be inert and invisible for production callers).
- Matrix: `ubuntu-22.04` + `ubuntu-24.04` (az/jq/openssl drift), plus one **minimal-container
  job** proving the declared dependency list is complete (catches "works only because the hosted
  image has X").

### 9.6 PR preview refs — the no-manual-intervention mechanism

Replaces `github-actions-terraform`'s documented find/replace + force-tag ritual. On every PR
(open/synchronize, **same-repo branches only** — fork PRs get tests but no preview, since the
job needs `contents: write`), `pr-preview.yml`:

1. Checks out the PR head.
2. Rewrites every internal annotated ref (`@vX.Y.Z # x-release-please-version` lines in
   `.github/workflows/reusable-*.yml`) to `@preview/pr-<N>`.
3. Creates a **detached generated commit** of that tree (`git commit-tree`, parent = PR head SHA
   for provenance) — the PR branch itself is never touched.
4. Force-pushes tag **`preview/pr-<N>`** at the generated commit.
5. Upserts a sticky PR comment:

   > **Test this PR from a calling repo:**
   > `uses: dsb-norge/cert-warden/.github/workflows/reusable-warden.yml@preview/pr-42`
   > (actions: `dsb-norge/cert-warden/actions/warden@preview/pr-42`) — refreshed on every push;
   > deleted when the PR closes.

6. On PR close/merge, a `pull_request: closed` job deletes the tag.

Properties: zero manual steps; internal refs in the preview are self-consistent (workflow and
actions resolve through the same tag); **nothing ever lands on main** — main only ever carries
exact-pinned release refs maintained by release-please. The tag is plain (no Release object),
excluded from the `v*` ruleset, and world-readable (public repo) so any calling repo can consume
it immediately.

### 9.7 L3b — remote-fetch e2e via the preview tag

Same-repo `uses: ./` doesn't exercise GitHub's *remote* fetch path (full-repo archive download,
`GITHUB_ACTION_PATH` resolution, the `lib/` sourcing claim from §3). So after pushing the
preview tag, CI dispatches **`preview-consume.yml` at that tag ref**
(`gh workflow run preview-consume.yml --ref preview/pr-<N>`): the workflow file inside the
generated commit already has its refs rewritten to `preview/pr-<N>`, so its jobs consume the
suite **as a genuine remote caller would** — running the harness-mode scenarios end-to-end. The
dispatching job waits and reports the conclusion into `ci-conclusion`. This is the automated
replacement for "go to a calling repo and try it" for everything except real Azure. (`uses:`
refs are static strings — dispatching a workflow *at the tag* is what makes the dynamic ref
possible.)

### 9.8 Coverage — self-contained `jq` gate (decided)

**kcov** over the bats suites (`--include-path=actions/,lib/`), Cobertura + JSON out; the gate
is a plain **`jq .percent_covered` threshold step** — no external coverage service (decided; no
Codecov). Start ~80% with headroom, **advisory at first** (warns, doesn't fail) because kcov's
bash parser has known heredoc blind spots and a reported bats-in-GHA nondeterminism; promote to
required once observed stable. The number lands in the step summary and (optionally) the sticky
CI comment for visibility. Coverage is a trend signal, not a precision gate — the *scenario
list* in §9.3 is the real completeness instrument.

### 9.9 Workflow hardening defaults (every workflow in the repo)

`permissions: {}` at workflow level with per-job grants; `defaults.run.shell: bash` everywhere
(never rely on the no-`pipefail` implicit default); `actions/checkout` with
`persist-credentials: false`; all third-party `uses:` SHA-pinned with version comments;
`.gitattributes` forces LF.

**Adopted (decided, was OQ-10): OpenSSF Scorecard + `step-security/harden-runner`**, on the
explicit condition that maintainers get thorough documentation — that's `docs/security-tooling.md`
(§10.3). Rollout: Scorecard as the standard weekly action with results in code scanning (badge
optional); harden-runner in **audit mode first** on all jobs, collecting egress baselines, then
`egress-policy: block` with per-job allow-lists once the baselines are stable (the CI jobs have
small, knowable egress: GitHub, GHCR for Pebble images, the Go module proxy for lego/pebble
installs).

## 12. Security posture (public repo)

- No secrets anywhere in the suite (OIDC-only in consumers; the repo's CI needs no Azure at

- Fork PRs: full read-only CI, no preview tag, no secrets exposure (standard `pull_request`
  semantics); `pull_request_target` is not used anywhere.
- Hardening defaults per §9.9; zizmor SARIF in code scanning; immutable releases + tag ruleset

- **No links to private DSB repos anywhere in the public repo — ever** (owner-mandated, hard

  generated changelog. Requirements are always expressed with self-contained examples (§7.3),

## Appendix A — runner/bash pitfalls catalog (seed for `docs/testing.md`)

Each entry ships with a named regression test where testable. Sources: GitHub docs, bats-core
gotchas, BashFAQ/105, incidents in this suite and `github-actions-terraform`.

| # | Pitfall | Mechanism / note |
|---|---|---|
| P-1 | `((count++))` aborts under `set -e` when count=0 | arithmetic result 0 ⇒ exit 1; use `x=$((x+1))`. **Already bit this suite** (metrics-loss incident); test ported from `selftest.sh` |
| P-2 | `$( )` doesn't inherit errexit | child dies, parent continues; fix `shopt -s inherit_errexit`; probe must run in a child *process*, not command substitution |
| P-3 | `local var=$(cmd)` masks `cmd`'s failure | `local`'s exit code wins; declare and assign separately |
| P-4 | errexit disabled inside any function used as a condition (`if f`, `f \|\| true`) | whole function body runs un-guarded; no fix — don't rely on `-e` inside such functions |
| P-5 | implicit shell ≠ `shell: bash` | unspecified Linux shell = `bash -e` **without pipefail**; always declare `shell: bash` / `defaults.run.shell` |
| P-6 | `--noprofile --norc` | no rc/profile env on self-hosted runners; never assume interactive-shell PATH |
| P-7 | pipefail × `grep` no-match | exit 1 is a valid "none found"; handle explicitly |
| P-8 | step outcome = last command's exit code | trailing `\|\| log-warn` silently greens a failed step |
| P-9 | `set -o allexport` + large data → ARG_MAX/128 KiB-per-var | exec fails exit 126 "argument list too long"; big payloads to `$RUNNER_TEMP` files. **Recurring production-only bug class in `github-actions-terraform`** |
| P-10 | multiline `GITHUB_OUTPUT` delimiter injection | value containing the delimiter forges outputs (cf. CVE-2022-35954); random delimiters mandatory |
| P-11 | workflow-command injection via echoed untrusted text | `::command::` lines in stdout are executed; `::stop-commands::` guards + `::add-mask::` before first print |
| P-12 | `${{ }}` interpolation into `run:` bodies | template injection; env-var indirection only (zizmor enforces) |
| P-13 | composite actions: no `INPUT_*` autoexport, `shell:` mandatory per step | explicit `env:` mapping in every shim |
| P-14 | CRLF via Windows-side edits | `$'\r': command not found`; `.gitattributes` `*.sh text eol=lf` |
| P-15 | bats: `run` never fails the test by itself; `run cmd \| grep` pipes the wrapper | assert on `$status`/`output`; wrap pipelines in functions |
| P-16 | bats: background process inheriting fd 3 hangs the suite | `close_non_std_fds` for daemons (Pebble, sink) |
| P-17 | GITHUB_ENV leaks to all later steps | never write secrets/large data; prefer `$RUNNER_TEMP` |
| P-18 | hosted-image tool drift (az/jq/openssl versions) | OS-matrix + minimal-container self-test job (§9.5) |
| P-19 | `actions/cache` key collisions across arch/os for binaries | cache key must include version+os+arch (`setup-lego`) |
| P-20 | Pebble non-determinism is a feature | 5% nonce rejects, 0-15 s VA sleeps, 50% authz reuse — pin `PEBBLE_VA_NOSLEEP=1`, `NONCEREJECT=0` for determinism, and keep one chaos test at `NONCEREJECT=20` |
